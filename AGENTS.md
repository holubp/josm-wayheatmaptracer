# AGENTS

Repository-specific guardrails for future changes:

- Treat JOSM coordinate spaces explicitly. Downloaded-area checks use `Bounds` with geographic coordinates. The current sliding core intentionally uses the 0.2.0-style rendered visible layer: candidates are tracked in oversampled screen space, projected back through `MapView`, and then carried as `EastNorth` only for preview/debug export.
- The downloaded-area bypass is a heatmap-only drawing escape hatch and must remain opt-in. Do not make live alignment skip `Bounds` checks by default.
- Junction and endpoint movement must remain opt-in. When it is enabled in precise mode, do not simplify away selected/shared anchor nodes.
- Do not reintroduce uniform point redistribution after simplification in precise mode. Straight sections should be allowed to collapse more aggressively than curves.
- When precise mode drops interior nodes, clean up only nodes that are both untagged and unreferenced. Shared or tagged nodes must survive.
- Keep verbose/debug logging useful for remote diagnosis. For alignment bugs, log selected segment ids, exact redacted settings, sampled color schemes, sampled bands, chosen candidate offsets, candidate evidence/SNR, and applied node moves.
- Palette changes must come with regression coverage in `HeatmapFixtureArchiveTest` or `RidgeTrackerTest`, especially for `blue`, `gray`, and internal multi-color/dual-color detection, which are easy modes to mis-rank.
- Palette evidence is semantic, not always raw brightness. Keep `hot`/`blue`/`purple` single-ramp ordering separate from `bluered` and `gray`, where hue/saturation identify high-activity centers.
- Keep the visible-layer alignment path close to 0.2.0 unless the maintainer explicitly asks to reintroduce fixed-tile inference. No-signal profiles use zero-offset fallback peaks, seeds come from the first informative profiles, and long unsupported runs keep the last state just as 0.2.0 did.
- Multi-color detection may show independent classifiers and 0.9.x combined-intensity classifiers on the same rendered visible layer. Combined classifiers must fuse named color-to-intensity mappings before peak extraction and ridge tracking. Do not add post-hoc geometry consensus, parallel-way scoring, fixed-tile validation, or internal refinement without explicit approval.
- Direct intensity source modes bypass palette mapping and read rendered pixel luminance, max channel, or alpha as scalar intensity. Keep `Color mapping` as the default for Strava PNG heatmaps; do not claim or depend on a Strava raw numeric heat API unless it is documented and permitted.
- Experimental detector variants such as `hot-corridor`, `bluered-cool`, `bluered-corridor`, `dual-corridor`, `gray-magenta`, `gray-corridor`, `gray-strict`, and `purple-strict` are for visual rating and calibration. They may influence default candidate ordering only through explicit calibrated ranking, and raw detector scores must remain exported so user-rated examples can be compared against the old candidates.
- Managed Strava settings create/refresh the visible TMS layer; live sliding does not sample fixed source tiles directly. Rendered colors come from the current visible layer, while cross-section half-width/step should stay ground-scale stable by converting reference-view pixels to effective current-view pixels. The preview/debug output should record the visible tile zoom and effective sampling parameters.
- The preview dialog is the place for ridge selection. Keep it modeless so users can pan/zoom and toggle layers while previewing. Keep candidate labels user-readable and avoid exposing raw internal scores as the primary UI.
- Candidate selection/rating after preview pan or zoom must use slide-time projected `EastNorth` candidate geometry. Never reproject stored raster/screen candidate points through the current `MapView` unless no slide-time geometry exists.
- Last-slide debug export should prioritize the most recent slide attempt and include the rendered heatmap capture, redacted settings, sampled colors, original/preview/candidate geometry, candidate evidence, source tile zoom, viewport bounds, effective view/raster meters per pixel, configured/effective cross-section width and step, estimated visible tile ranges, per-detector cross-section peak data, candidate/profile/palette CSV calibration exports, and per-slide logs. Never export cookies, signed headers, or full signed URLs.
- Cross-section peak diagnostics should preserve gradient strength/balance and candidate evidence should preserve mean gradient strength plus longitudinal stability; these metrics help distinguish true ridges from noisy intensity-only maxima.
- Keep `scripts/analyze-debug-bundles.py` compatible with exported bundles when changing candidate metrics, ratings, or CSV diagnostics.
- Candidate ratings and negative feature tags from the opt-in preview rating mode are calibration data. Preserve them in `candidate-ratings.json`, `status.json`, and verbose log lines when changing diagnostics export behavior.
- Rough full-way 2-5 node selections may be recognized in UI/debug metadata, but the current visible-layer path keeps the configured alignment mode to preserve predictable sliding behavior.
- Keep the fixture regression acceptance envelope and `acceptable-limits.osm` generation tied to the same configured metric radius. Visual limit changes must be reflected in the regression oracle too.
- `acceptable-limits.osm` is a regression/visualization artifact only. Do not reuse its envelope logic inside the live alignment workflow.
- Release versioning must match the scope of change. Stay on `0.x.x` until the maintainer declares the plugin suitable for broader use. After `1.0.0`, use `1.x.x` for major functionality or architecture changes, `1.1.x`-style minor releases for smaller features and improvements, and `1.1.1`-style patch releases only for bug fixes. Release naming follows the JOSM AudioWptMarker convention: tag/release name `v<version>`, primary jar asset `wayheatmaptracer.jar`, and semantic version in the jar manifest `Plugin-Version`.
