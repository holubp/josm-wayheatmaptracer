# AGENTS

Repository-specific guardrails for future changes:

- Treat JOSM coordinate spaces explicitly. Downloaded-area checks use `Bounds` with geographic coordinates. The visible-layer fallback intentionally uses the 0.2.0-style rendered layer: candidates are tracked in oversampled capture space, projected back through the slide-time capture bounds, and then carried as `EastNorth` only for preview/debug export.
- Visible-layer alignment must render the selected segment extent at the required working resolution, using chunked virtual viewport captures when a single viewport would be too large. It must fail clearly when selected-segment profiles still fall outside the captured raster. Off-screen fallback profiles must not silently pull endpoints toward zero offset or lower confidence as if the heatmap itself were weak.
- The downloaded-area bypass is a heatmap-only drawing escape hatch and must remain opt-in. Do not make live alignment skip `Bounds` checks by default.
- Junction and endpoint movement must remain opt-in. When it is enabled in precise mode, do not simplify away selected/shared anchor nodes.
- Do not reintroduce uniform point redistribution after simplification in precise mode. Straight sections should be allowed to collapse more aggressively than curves.
- When precise mode drops interior nodes, clean up only nodes that are both untagged and unreferenced. Shared or tagged nodes must survive.
- Keep verbose/debug logging useful for remote diagnosis. For alignment bugs, log selected segment ids, exact redacted settings, sampled color schemes, sampled bands, chosen candidate offsets, candidate evidence/SNR, and applied node moves.
- Palette changes must come with regression coverage in `HeatmapFixtureArchiveTest` or `RidgeTrackerTest`, especially for `blue`, `gray`, and internal multi-color/dual-color detection, which are easy modes to mis-rank.
- Palette evidence is semantic, not always raw brightness. Keep `hot`/`blue`/`purple` single-ramp ordering separate from `bluered` and `gray`, where hue/saturation identify high-activity centers.
- Broad saturated corridors must be centered on the high-intensity core, not on the whole low-threshold shoulder. Add/update sampler and tracker regression tests when changing plateau thresholds or longitudinal smoothing, and preserve real sustained switchbacks while suppressing one-profile shoulder aliasing.
- For visible/source `bluered`, keep default candidate ordering biased toward `bluered-combined`, `bluered-corridor`, `bluered-cool`, and `bluered`. Generic brightness/corridor detectors such as `hot`, `hot-corridor`, `dual-corridor`, and `multi-combined` may be useful for exported calibration/manual preview, but should not outrank native bluered semantics by default because they can oscillate between corridor shoulders.
- Keep the visible-layer alignment path close to 0.2.0 unless the maintainer explicitly asks to reintroduce fixed-tile inference. No-signal profiles use zero-offset fallback peaks, seeds come from the first informative profiles, and long unsupported runs keep the last state just as 0.2.0 did. True no-signal candidates may be exported for diagnostics, but must not be applicable.
- Multi-color detection may show independent classifiers and 0.9.x combined-intensity classifiers on the same sampled color source, either managed source tiles or the legacy visible rendered layer. Combined classifiers must fuse named color-to-intensity mappings before peak extraction and ridge tracking. Do not add post-hoc geometry consensus, parallel-way scoring, separate fixed-tile validation, or internal refinement without explicit approval.
- Direct intensity source modes bypass palette mapping and read rendered pixel luminance, max channel, or alpha as scalar intensity. Keep `Color mapping` as the default for Strava PNG heatmaps; do not claim or depend on a Strava raw numeric heat API unless it is documented and permitted.
- Experimental detector variants such as `hot-corridor`, `bluered-cool`, `bluered-corridor`, `dual-corridor`, `gray-magenta`, `gray-corridor`, `gray-strict`, and `purple-strict` are for visual rating and calibration. They may influence default candidate ordering only through explicit calibrated ranking, and raw detector scores must remain exported so user-rated examples can be compared against the old candidates.
- Managed Strava settings create/refresh the visible TMS layer and, when all access values are configured, live sliding samples managed source tiles directly for fixed-resolution alignment. Without managed access values, the legacy visible-layer path remains the fallback; rendered colors then come from JOSM's normal layer rendering captured over the selected extent at the working resolution. The preview/debug output should record whether sampling used managed source tiles or the visible rendered layer, plus source tile zoom, capture bounds, chunk count, and effective sampling parameters.
- Stable fixed-scale sampling must not use broad max-dilation at the default z15 inference zoom; it can flatten the cross-section gradient and make off-center ways barely move. Keep any lower-zoom dilation small and covered by tests.
- The preview dialog is the place for ridge selection. Keep it modeless so users can pan/zoom and toggle layers while previewing. Keep candidate labels user-readable and avoid exposing raw internal scores as the primary UI.
- Candidate selection/rating after preview pan or zoom must use slide-time projected `EastNorth` candidate geometry. Never reproject stored raster/screen candidate points through the current `MapView` unless no slide-time geometry exists.
- Modeless preview apply must be guarded against stale source geometry. Revalidate the way node sequence, selected segment identities, dataset membership, and source coordinates before candidate switching and before apply.
- Repeated-node selections are unsafe unless occurrence identity is explicitly modeled. Reject selected segments where a segment node appears more than once in the same way, including outside the selected range.
- Last-slide debug export should prioritize the most recent slide attempt and include the rendered heatmap capture, redacted settings, sampled colors, original/preview/candidate geometry, candidate evidence, source tile zoom, viewport bounds, effective view/raster meters per pixel, configured/effective cross-section width and step, estimated visible tile ranges, per-detector cross-section peak data, candidate/profile/palette CSV calibration exports, and per-slide logs. Never export cookies, signed headers, or full signed URLs.
- Verbose logs should also include the full redacted alignment settings JSON, including inference mode (`STABLE_FIXED_SCALE` vs `RAW_HIGH_RESOLUTION`), inference/validation zooms, intensity source, and cache-buster state, so console-only reports can still identify the active sampling mode.
- Cross-section peak diagnostics should preserve gradient strength/balance and candidate evidence should preserve mean gradient strength plus longitudinal stability; these metrics help distinguish true ridges from noisy intensity-only maxima.
- Keep `scripts/analyze-debug-bundles.py` compatible with exported bundles when changing candidate metrics, ratings, or CSV diagnostics.
- Candidate ratings and negative feature tags from the opt-in preview rating mode are calibration data. Preserve them in `candidate-ratings.json`, `status.json`, and verbose log lines when changing diagnostics export behavior.
- Rough full-way 2-5 node selections may be recognized in UI/debug metadata, but the current visible-layer path keeps the configured alignment mode to preserve predictable sliding behavior.
- Rough full-way 2-5 node selections must not silently widen managed source-tile search beyond the configured search half-width. If a mapper needs a wider rough-sketch search, it must be explicit in settings; automatic widening can jump to unrelated parallel traces.
- Candidates with abrupt lateral jumps or acceleration are structurally unsafe even when heatmap SNR is high. Keep candidate safety warnings exported and block preview/apply for those candidates rather than relying only on signal strength.
- Precise-shape simplification must run after fixed anchors are restored and per fixed-anchor interval. Do not simplify the whole traced centerline before fixed-anchor interval reconstruction.
- Precise-shape undo/redo must replay stored target coordinates for reused existing nodes on every command execute/redo, not only during first replacement-node construction.
- Keep the fixture regression acceptance envelope and `acceptable-limits.osm` generation tied to the same configured metric radius. Visual limit changes must be reflected in the regression oracle too.
- `acceptable-limits.osm` is a regression/visualization artifact only. Do not reuse its envelope logic inside the live alignment workflow.
- Release versioning must match the scope of change. Stay on `0.x.x` until the maintainer declares the plugin suitable for broader use. After `1.0.0`, use `1.x.x` for major functionality or architecture changes, `1.1.x`-style minor releases for smaller features and improvements, and `1.1.1`-style patch releases only for bug fixes. Release naming follows the JOSM AudioWptMarker convention: tag/release name `v<version>`, primary jar asset `wayheatmaptracer.jar`, and semantic version in the jar manifest `Plugin-Version`.
